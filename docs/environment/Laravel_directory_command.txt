------------------------------------------------------------------------
 0. Database / Model [土台となるデータ構造]
------------------------------------------------------------------------
- app/Models
  コマンド：php artisan make:model [Name]

  用途：・データベースのテーブルと1対1で対応するクラスを定義する。
        ・テーブル間の関係（リレーション）や、データの型変換を記述する。
        ・データの検索、保存、更新などのEloquent操作の窓口となる。

  例  ：・Userモデルで、投稿（Posts）との一対多のリレーションを定義。
        ・Productモデルで、価格カラムを自動で数値型にキャストする設定。
        ・Orderモデルで、特定の期間内の注文だけを絞り込むスコープを作成。

- database/factories
  コマンド：php artisan make:factory [Name]Factory --model=[ModelName]

  用途：・テストや開発環境で使う「ダミーデータ」の生成ルールを定義する。
        ・Fakerライブラリを使い、本物に近い名前や住所を自動生成させる。
        ・テストコード内で、100件のデータを1行で作成するために使用する。

  例  ：・ユーザー登録テスト用に、ランダムな名前とメアドを持つデータを生成。
        ・商品一覧表示の確認用に、適当な価格と説明文を持つ商品を大量作成。
        ・期限切れクーポンなど、特定の状態を持つデータを意図的に作成。

- database/seeders
  コマンド：php artisan make:seeder [Name]Seeder

  用途：・開発環境や本番環境に、あらかじめ必要な「初期データ」を投入する。
        ・Factoryと組み合わせて、数千件のテストデータを一括で流し込む。
        ・マスターデータ（都道府県、カテゴリー、基本設定）を固定で登録する。

  例  ：・開発を始める際、ログイン可能な「テスト用管理者アカウント」を作成。
        ・ECサイトの構築時に、あらかじめ決まった「商品カテゴリ名」を登録。
        ・デモ環境用に、昨日の日付で作成された注文データを100件作成。

------------------------------------------------------------------------
 1. Logic / Events / Queue [連動・非同期処理]
------------------------------------------------------------------------
- app/Events
  コマンド：php artisan make:event [Name]

  用途：・システム内で発生した「出来事」を名前をつけて定義する。
        ・その出来事に付随するデータ（モデル等）を保持するのみに徹する。
        ・処理ロジックは書かず、Listenersに渡す「伝票」の役割を果たす。

  例  ：・注文完了イベントに、購入した商品リストとユーザーIDを保持させる。
        ・支払い失敗イベントに、エラーコードと決済試行日時をセットする。
        ・会員登録イベントに、登録されたUserモデルそのものを詰め込む。

- app/Listeners
  コマンド：php artisan make:listener [Name]

  用途：・イベント放送をキャッチして、実際の「実務処理」を記述する場所。
        ・1つのイベントに対して、複数の異なる実務担当（クラス）を配置する。
        ・重い処理（メール送信等）をキューへ送るかどうかの判断もここで行う。

  例  ：・注文完了イベントを受け取り、在庫DBの数字を減らす処理を実行。
        ・注文完了イベントを同時に受け取り、サンクスメールを送信する。
        ・支払い失敗イベントを受け取り、管理者へSlackで警告を飛ばす。

- app/Jobs
  コマンド：php artisan make:job [Name]

  用途：・ユーザーを待たせないよう、裏側で実行する重い処理を担う。
        ・失敗時のリトライ回数や、実行までの遅延時間を細かく制御する。
        ・外部API通信など、成否が不安定な処理をメインから切り離す。

  例  ：・登録済みユーザー1万人に対する、新製品メルマガの一斉送信処理。
        ・ユーザーがアップロードした巨大な動画ファイルの変換・圧縮作業。
        ・外部の会計ソフトや配送管理システムへのバッチ的なデータ転送。

- app/Observers
  コマンド：php artisan make:observer [Name]

  用途：・モデルのライフサイクル（保存・削除等）を常に監視する。
        ・DB操作が発生した際に、必ずセットで動かすべき自動処理を行う。
        ・どこからモデルが操作されても、データの整合性を強制的に守る。

  例  ：・ユーザー削除時に、関連するプロフィール画像をサーバーから消す。
        ・記事の保存時に、タイトルからURL用のスラッグを自動生成する。
        ・投稿作成時に、その投稿者の「最終アクティブ日時」を更新する。

------------------------------------------------------------------------
 2. HTTP / API / Validation [入り口の制御]
------------------------------------------------------------------------
- routes/ [api.php / web.php]
  コマンド：なし（手動で追記）

  用途：・URLと、それに対応するコントローラのアクションを紐づける。
        ・外部API用(api.php)とブラウザ表示用(web.php)で入口を分ける。
        ・ミドルウェア（認証など）をどのURLに適用するかを管理する。

  例  ：・「GET /users」が来たら、UserControllerのindexメソッドを呼ぶ。
        ・「/admin」で始まる全てのURLに、管理者認証ミドルウェアを適用。
        ・API用のURLには、自動的に「/api」というプレフィックスを付ける。

- app/Http/Middleware
  コマンド：php artisan make:middleware [Name]

  用途：・処理がアプリの中核に届く前に、共通の「フィルター」を通す。
        ・不正なリクエストの門前払いや、データの書き換えを担う。
        ・認証やログ記録など、多くのルートで使い回したい壁を作る。

  例  ：・ログインしていないユーザーを、ログイン画面へ強制移動させる。
        ・管理者以外が管理者用URLを叩いた際、403エラーを返して遮断。
        ・入力された全データから、不要な前後の空白を自動で削除する。

- app/Http/Requests
  コマンド：php artisan make:request [Name]

  用途：・コントローラが動く前に、入力データの正当性を100%保証する。
        ・バリデーション（形式チェック）と、簡単な権限確認を担う。
        ・複雑なチェックロジックをコントローラから排除し、見通しを良くする。

  例  ：・パスワード再設定時に、新旧パスワードが同じでないか判定する。
        ・アップロードされたファイルが、指定の画像形式か厳密にチェック。
        ・編集リクエストを送った人が、そのデータの編集権限を持つか確認。

- app/Http/Resources
  コマンド：php artisan make:resource [Name]

  用途：・データベースの生データを、外部公開用のJSON形式に整形する。
        ・フロントエンドやアプリ側が扱いやすい構造へデータを変換する。
        ・DBのカラム名を隠蔽し、システムの内部構造を外部に漏らさない。

  例  ：・姓と名を結合して「氏名」にしたり、生年月日から「年齢」を作る。
        ・内部的なIDや管理用フラグを、レスポンスから除外して隠す。
        ・日時データを「2026/04/27」のように、指定の表示形式に揃える。

- app/Rules
  コマンド：php artisan make:rule [Name]

  用途：・標準機能にはない、そのアプリ独自の複雑なチェックルールを作る。
        ・複数のプロジェクトや箇所で使い回せる「検証部品」として定義。
        ・外部APIへの問い合わせが必要な、高度なバリデーションを行う。

  例  ：・「過去に一度も使ったことがないパスワードか」をDBと照合。
        ・入力された郵便番号が、外部の住所検索サービスに実在するか確認。
        ・特定の禁止ワードリストに基づき、不適切な表現がないかチェック。

------------------------------------------------------------------------
 3. Mail / Notifications / UI [通知・出力]
------------------------------------------------------------------------
- app/Mail
  コマンド：php artisan make:mail [Name]

  用途：・送信メールの「本文デザイン」と「送信設定」を一箇所にまとめる。
        ・HTMLメールとテキストメールの出し分けを自動的に管理する。
        ・添付ファイルやインライン画像の埋め込み設定をコード化する。

  例  ：・決済完了通知メールに、PDF形式の領収書を自動で添付して送る。
        ・ユーザーの言語設定に応じて、英語・日本語の本文を切り替える。
        ・会社のロゴ画像が含まれた、リッチなHTMLデザインのメルマガ構築。

- app/Notifications
  コマンド：php artisan make:notification [Name]

  用途：・1つの通知をメール・Slack・DB・LINE等へ横断的に飛ばす。
        ・ユーザーごとに「どの媒体で通知を受け取るか」を柔軟に選ばせる。
        ・通知の履歴をDBに保存し、ログイン後の「お知らせ一覧」を作る。

  例  ：・重大なシステムエラーを、開発者のSlackとメールへ一斉送信。
        ・SNSで「いいね」された際、スマホへの通知と画面内の赤い点を作る。
        ・注文状況の更新を、メールとマイページの通知履歴の両方に残す。

- app/View/Components
  コマンド：php artisan make:component [Name]

  用途：・UIパーツの「HTML（見た目）」と「表示ロジック」をセットにする。
        ・複数の画面で使う共通部品を、一つのタグとして再利用可能にする。
        ・Bladeの中に複雑なif文を書かず、クラス側で表示判断を行う。

  例  ：・ログイン状態や権限で中身が変わる「共通ヘッダー」の作成。
        ・特定のデータ（記事や商品）をカード形式で表示する共通パーツ。
        ・ページごとにタイトルやメタタグを動的に変更する部品。

------------------------------------------------------------------------
 4. Security / Customization [高度な設定]
------------------------------------------------------------------------
- app/Policies
  コマンド：php artisan make:policy [Name]

  用途：・「このデータ」に対し「この人」が何をして良いかを判定する。
        ・認可のルールをモデルと1対1で管理し、権限周りの混乱を防ぐ。
        ・Laravelの命名規則に従い、自動的にモデルと紐づいて動作する。

  例  ：・「自分の書いた記事」だけが編集可能で、他人のものは閲覧のみ。
        ・「管理者」であれば、どのユーザーの投稿でも強制的に削除できる。
        ・「無料会員」は動画を3本まで、「有料会員」は無制限に視聴可能。

- app/Console/Commands
  コマンド：php artisan make:command [Name]

  用途：・黒い画面（Artisan）から直接実行できる自作ツールを開発する。
        ・定期実行（Cron）と連携し、サーバー側の保守業務を自動化する。
        ・開発中に必要な一括データ処理や、初期設定をコマンド化する。

  例  ：・毎日深夜3時に、期限切れの無料クーポンをDBから自動で無効化。
        ・1週間以上ログインのない「仮登録ユーザー」を自動で削除する。
        ・外部の天気予報APIを叩き、最新の予報データを自社DBに取り込む。

- app/Exceptions
  コマンド：php artisan make:exception [Name]

  用途：・エラー発生時の挙動（報告や画面表示）を独自に定義する。
        ・特定の業務エラーが発生した際に、専用のレスポンスを返す。
        ・ログ出力のカスタマイズや、外部への通知ロジックを統合する。

  例  ：・在庫不足エラー時に、専用の購入失敗画面とアラートを表示。
        ・外部APIのダウン時に、管理者へSlackで緊急事態を報告する。
        ・特定の条件下で発生する404エラーを、別のURLへリダイレクト。

------------------------------------------------------------------------
 5. Manual Directories [設計上の工夫] ※手動作成
------------------------------------------------------------------------
- app/Services
  コマンド：なし（手動で作成）

  用途：・複数のモデルが絡む、複雑な「業務の手順（ロジック）」をまとめる。
        ・コントローラに書くと長くなりすぎる処理を、外に切り出す。
        ・他のコントローラやジョブからも、同じ手順を再利用可能にする。

  例  ：・「カート計算・クーポン適用・在庫確保・決済」の一連の購入フロー。
        ・複数の画像サイズを作成し、それぞれ別々のフォルダに保存する処理。
        ・ユーザーのポイント計算やランクアップ判定の複雑なアルゴリズム。

- app/DTOs (Data Transfer Objects)
  コマンド：なし（手動で作成）

  用途：・データを運ぶための「決まった形の箱」を作り、型を安全にする。
        ・配列（Array）の代わりに使い、中身に何が入っているか保証する。
        ・メソッド間で大量の情報を渡す際に、引数をスッキリまとめる。

  例  ：・会員登録フォームの大量の項目を、UserDataクラスにまとめて運ぶ。
        ・外部APIから返ってきたバラバラな情報を、固定の形式に整理する。
        ・コントローラからサービスへ、必要なデータだけを抽出して渡す。

------------------------------------------------------------------------
 6. 実装の迷いを解消する判断基準 (Usage Decisions)
------------------------------------------------------------------------
■ Observers or Events & Listeners [自動か、連鎖か]
  - Observers：DBの値が変わったら「必ず」セットで行うべき単純な処理。
  - Events & Listeners：一つの出来事から複数の外部処理が「連鎖」する時。

■ Jobs or Commands [裏側か、定時か]
  - Jobs：ユーザー操作を起点に、待たせず「裏側」ですぐ実行したい時。
  - Commands：ユーザー操作とは無関係に「定時」で実行したい時。

■ Policies or Gates [モデルか、機能か]
  - Policies：判定対象が「特定のデータ（モデル）」である場合。
  - Gates：判定対象がモデルに依存しない「システム全体の機能」の場合。

■ Services or Repositories [ロジックか、クエリか]
  - Services：「どういう手順か」というビジネス上のルールを書きたい時。
  - Repositories：「どうデータを取得するか」という複雑なSQLを隠したい時。

■ Notifications or Mail [多機能か、メール特化か]
  - Notifications：通知が目的で、Slackなど複数の手段を使い分けたい時。
  - Mail：文書を送ることが目的で、詳細なレイアウトや添付が必要な時。

------------------------------------------------------------------------
 7. データの流れ（推奨されるクリーンな循環）
------------------------------------------------------------------------
 ・[Route]    -> 入口のURL判定とミドルウェア適用
 ・[Request]  -> 入力値のバリデーションと簡易的な認可
 ・[Policy]   -> その操作を行って良いか厳密に判定
 ・[Service]  -> 実際の計算や複雑なビジネスロジックの実行
 ・[Model]    -> データベースへの保存
 ・[Observer] -> 保存に付随するDB的な自動メンテナンス
 ・[Event]    -> 「完了した」という事実をシステム全体へ放送
 ・[Listener] -> 各担当者がメール送信やJobの投入を開始
